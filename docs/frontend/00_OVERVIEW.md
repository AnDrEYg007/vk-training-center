# 0. Обзор фронтенда (React)

Этот документ предоставляет высокоуровневый обзор архитектуры, технологий и ключевых принципов, на которых построено фронтенд-приложение.

## 1. Назначение и ключевые задачи

Фронтенд является клиентской частью приложения, с которой непосредственно взаимодействует пользователь. Его основные задачи:

1.  **Отображение пользовательского интерфейса**: Рендеринг всех визуальных компонентов, таких как календарь, сайдбар, модальные окна и карточки постов.
2.  **Обработка действий пользователя**: Реагирование на клики, перетаскивания, ввод текста и другие взаимодействия.
3.  **Управление состоянием UI**: Отслеживание того, какой проект выбран, какая вкладка активна, какие модальные окна открыты.
4.  **Взаимодействие с бэкендом**: Отправка запросов на бэкенд для получения, сохранения или удаления данных и последующее обновление интерфейса на основе полученного ответа.

## 2. Технологический стек

-   **React**: Основа приложения, библиотека для создания пользовательских интерфейсов.
-   **TypeScript**: Статическая типизация для повышения надежности и упрощения разработки.
-   **Tailwind CSS**: Утилитарный CSS-фреймворк для быстрой и консистентной стилизации компонентов.
-   **Без сборщика (No-Bundler Setup)**: Для упрощения локальной разработки проект не использует Webpack или Vite. Он полагается на нативные ES-модули и `import maps`, которые поддерживаются современными браузерами. Зависимости подгружаются напрямую из CDN.

## 3. Основные принципы архитектуры

-   **Feature-Sliced Design (FSD)**: Код организован по "фичам" (бизнес-сущностям), что делает структуру проекта понятной и масштабируемой.
-   **Централизованное управление состоянием данных**: Вся логика, связанная с получением, хранением и обновлением данных с сервера, вынесена в единый `ProjectsContext`, что устраняет дублирование и упрощает отладку.
-   **Четкое разделение ответственности**:
    -   **Компоненты** отвечают только за отображение и вызов действий.
    -   **Хуки** инкапсулируют сложную логику и управление состоянием.
    -   **Сервисы API** абстрагируют все сетевые взаимодействия.
-   **Композиция компонентов**: Сложные UI-элементы (например, модальное окно поста) разбиты на множество мелких, переиспользуемых дочерних компонентов, каждый из которых решает одну конкретную задачу.
-   **Оптимистичный UI (где применимо)**: Интерфейс старается быть максимально отзывчивым, отображая данные из кэша немедленно, и только потом, при необходимости, подгружая обновления в фоне.
